This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Single-file web application (`index.html`) for managing beach volleyball (futevolei) tournaments for the CPMF club. No build step, no dependencies to install — open the file directly in a browser or deploy as-is.

## Deployment

Changes go live by pushing to the `main` branch of `NeanderTiso/CPMF` on GitHub. The file is served as a static page (GitHub Pages or similar).

```bash
git add index.html
git commit -m "..."
git push origin main
```

## Firebase Security Rules

The file `firestore.rules` is the **source of truth** for Firestore security rules. It is **not applied automatically** — this project has no Firebase CLI configured. Whenever the rules change, they must be copied manually to the Firebase Console:

**Firebase Console → Firestore Database → Rules → (colar o conteúdo de `firestore.rules`) → Publish**

### Rule summary per collection

| Collection | Read | Write |
|---|---|---|
| `tournaments` | Public | `players` field: public; everything else: admin only |
| `rankingConfig` | Public | Admin only (authenticated) |
| `mensalidades` | Public | Public (athletes confirm without login) |

### When to update `firestore.rules`

Every time a new Firestore collection is introduced in `index.html`, add its rule block to `firestore.rules` and remind the user to publish the updated rules in the Firebase Console.

## Architecture

Everything lives in `index.html`: HTML structure, CSS (inline `<style>`), and JavaScript (ES module `<script type="module">`).

### Backend: Firebase (no server)
- **Firestore** (`db`): collections in use:
  - `tournaments/{id}` — full tournament object, read/written via `saveTournament(t)` / `getTournament(id)`
  - `rankingConfig/manualPoints` — manual historical points for the overall ranking (`{ entries: [...] }`)
  - `mensalidades/{YYYY-MM}` — monthly fee payments per athlete (`{ month, payers: [{player, confirmedAt}] }`)
- **Auth**: email/password for admin login; anonymous sign-in for public users.
- Real-time public views use `onSnapshot`.

### State
A single `state` object tracks: `adminLoggedIn`, `currentTournamentId`, `unsubscribe` (Firestore listener).

### Pages (SPA)
Navigation is done by toggling `.page.active` via `showPage(id)`. Page IDs: `pageHome`, `pageAdminLogin`, `pageAdminDash`, `pageCreateTournament`, `pageAdminTournament`, `pagePublicTournaments`, `pagePlayerRegister`, `pagePublicView`, `pageRanking`, `pagePenalties`, `pageManualPoints`, `pageMensalidades`.

### Tournament Lifecycle
`status` field drives the flow:
1. `open` → registration open
2. `closed` → registration closed, phase 1 not yet started
3. `phase1` → round-robin matches generated by `generatePhase1Matches()`
4. `phase2` → top 8 advance to semifinals/final (generated by `generatePhase2()`)
5. `finished` → final results recorded

### Phase 1 Matching (`generatePhase1Matches`)
Each player plays 4 games. Rounds are generated iteratively. Within each group of 4 players, all 3 possible team splits are evaluated and the one with the fewest repeated pairs (tracked via `pairCount`) is chosen.

### Phase 1 Classification (`calcStandings`)
Sorted by: **wins** (descending), then **saldo** (score differential) as tiebreaker. No points column.

### Overall Ranking (`calcOverallRanking`)
Aggregates points across all `finished` tournaments for every player in `VALID_PLAYERS`. Scoring constants: `SCORE_CHAMPION`, `SCORE_RUNNER_UP`, `SCORE_THIRD`, `SCORE_FOURTH`, `SCORE_WIN`, `SCORE_PENALTY`.

### Registered Players
`VALID_PLAYERS` is a hardcoded array of allowed nicknames. Players can only register using names from this list.

## Key Conventions

- All CSS uses CSS custom properties defined in `:root` — dark/night theme is active. When changing colors, prefer updating variables over hardcoded hex values.
- Inline styles with hardcoded colors (e.g. `color:#666`) exist in JS-generated HTML strings — these must be updated manually when theming.
- JS is minified/compacted on single lines. Match the surrounding style when editing.
- `saveTournament(t)` overwrites the entire document — always fetch fresh with `getTournament(id)` before modifying.
